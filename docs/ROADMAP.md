# Roadmap проекта
## Mark of the Abyss

**Срок:** 6 месяцев

**Жанр:** 2D Action Platformer с боссами

**Технологии:** TypeScript, Canvas API, без библиотек

**Цель:** сделать полноценную игру с собственным runtime, базовыми инструментами и применением тем курса

***

### ЭТАП 1. Архитектура проекта и базовый runtime

###### Цели этапа

* поднять структуру проекта
* сделать базовый loop
* сделать базовый рендер
* организовать input
* организовать scene switching

###### Что делать

##### 1.1. Создать структуру проекта

```text
src/
  engine/
    core/
    renderer/
    input/
    scene/
    events/
    assets/
    math/

  game/
    player/
    enemy/
    boss/
    world/
    combat/
    abilities/
    progression/
    cutscene/
    ui/

  tools/
    level-editor/
    animation-editor/
    exporter/

  data/
    levels/
    animations/
    config/
    localization/
```

##### 1.2. Реализовать game loop

Нужен loop с фиксированным update step.

Нужно решить:

* использовать fixed timestep
* отделить update от render
* уметь паузить игру
* уметь ограничивать dt

##### 1.3. Создать базовый renderer

Сначала достаточно:

* очистка экрана
* drawRect
* drawImage
* drawText
* простая камера

##### 1.4. Реализовать input subsystem

Нужны:

* keyboard state
* key pressed / key released
* input buffering
* возможность делать action mapping

Пример action map:

* moveLeft
* moveRight
* jump
* attack
* skill1
* skill2
* demonToggle
* interact

##### 1.5. Сделать Scene Manager

Минимум:

* BootScene
* LoadingScene
* LevelScene
* CutsceneScene

Нужны:

* enter()
* exit()
* update()
* render()

##### 1.6. Сделать Event Bus

Минимальная реализация:

* subscribe
* unsubscribe
* emit

Это пригодится для:

* combat events
* animation events
* boss phase changes
* progression

############## Что проверить по окончании этапа

* приложение стабильно запускается
* есть цикл update/render
* есть переключение сцен
* работает клавиатура
* можно рисовать спрайт/прямоугольник на сцене

###### Результат этапа

Есть минимальный runtime, на котором уже можно строить игру.

### ЭТАП 2. Математика, карта и платформерное ядро

###### Цели этапа

* сделать движение персонажа
* реализовать физику
* реализовать tilemap
* реализовать коллизии
* реализовать камеру

###### Что делать

##### 2.1. Math module

Нужны:

* Vector2
* clamp
* lerp
* sign
* normalize
* distance
* rect intersection

##### 2.2. TileMap

Сразу делай карту не как массив объектов, а компактно.

Рекомендуемый подход:

* хранение в `Uint16Array`
* width, height
* функции getTile/setTile
* функция worldToTile
* функция tileToWorld

##### 2.3. Битовая упаковка тайлов

Пример:

* 0–9 бит: tile id
* 10 бит: solid
* 11 бит: hazard
* 12 бит: oneWay
* 13 бит: checkpoint
* 14 бит: decorative
* 15 бит: reserved

##### 2.4. Collision engine

Нужно реализовать:

* AABB
* проверку пересечения с tilemap
* разделение движения по X/Y
* разрешение коллизии
* grounded detection
* head collision
* wall collision

##### 2.5. Character Controller

Нужно:

* ускорение
* торможение
* прыжок
* падение
* гравитация
* max fall speed
* coyote time
* jump buffer

##### 2.6. Camera

Нужны:

* следование за игроком
* dead zone
* clamp по границам уровня
* плавность

############## Что проверить по окончании этапа

* персонаж бегает
* персонаж прыгает
* персонаж не проваливается
* персонаж корректно упирается в стены
* камера работает
* уровень читается из данных

###### Результат этапа

У тебя уже есть настоящий платформерный базис.

# 7. Риски проекта и как их контролировать

## Риск 1. Слишком большой объём контента

### Симптом

Ты делаешь уровни медленнее, чем ожидал.

####### Что делать

* сократить длину уровней
* уменьшить количество уникальных врагов
* объединить похожие сегменты
* не добавлять новый тип системы, пока не завершён core

---

## Риск 2. Боссы съедают слишком много времени

####### Что делать

* строить всех боссов на одном framework
* reuse телеграфов, фаз и шаблонов
* различать боссов комбинациями, а не уникальным кодом с нуля

---

## Риск 3. Редакторы мешают закончить игру

####### Что делать

* сначала полностью завершить core game
* editor делать только после working content pipeline
* editor не должен блокировать возможность делать уровни руками

---

## Риск 4. Binary format затягивает разработку

####### Что делать

* сначала JSON
* потом exporter в бинарь
* runtime должен уметь жить и с JSON, и с binary хотя бы на стадии отладки

---

## Риск 5. Плохой combat feel

####### Что делать

* заложить отдельное время на тюнинг
* регулярно тестировать игру руками
* не откладывать feel на последний день
